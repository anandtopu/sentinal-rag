## Decisions made during R2

1. **Two-tier verdict semantics: NULL vs `"skipped"`.** A persisted
   `NULL` means the layer never ran (flag off, or cascade short-
   circuited). `"skipped"` means the layer was requested but had no real
   backend / failed parse. Operators triaging "is the cascade healthy"
   need to see both, so the schema accepts both and the cascade
   distinguishes them deliberately.
2. **Adapters live in `sentinelrag_shared/evaluation/grounding/`, not
   the offline `evaluation/` package.** Reason: shape mismatch. The
   offline evaluators (ragas-style) take `EvalCase` + `EvalContext` with
   case_ids and rubrics; the online cascade takes
   `(answer, context_text)` and returns a verdict. The handoff guidance
   to "reuse, don't reimplement" assumed the NLI/judge evaluators
   existed — they didn't. Building them as a new package keeps the
   offline+online concerns from leaking into each other.
3. **Default adapters are NoOps everywhere.** The orchestrator builds
   the cascade with NoOpNli + NoOpJudge + StaticFeatureFlags, so a fresh
   deploy carries the cascade dormant until operators bind real
   adapters + raise the flags. No surprise cost from flipping a flag in
   isolation.
4. **Static flag client now, Unleash adapter as a follow-up.** Both
   live behind the same `FeatureFlagClient` Protocol so the swap is
   constructor-level. Settings already carry the Unleash URL +
   credentials (`apps/api/app/core/config.py`); the missing piece is the
   `UnleashClient`-backed implementation of `bool_flag` / `float_flag`.

## Open follow-ups surfaced during R2

- **Wire real adapters in the route handler.**
  `apps/api/app/api/v1/routes/query.py::execute_query` currently builds
  the orchestrator without `nli_backend` / `judge` / `flag_client`, so
  the cascade runs with NoOps in prod-shaped deploys too. A follow-up
  PR should:
  1. Build a `HuggingFaceNliBackend` (or `RetrievalServiceNliClient`
     that calls a small endpoint on the retrieval-service pod) and
     register it on `app.state` at startup.
  2. Build a `LiteLLMJudge` from a configurable judge-model alias and
     register similarly.
  3. Build an `UnleashFeatureFlags` adapter and register it.
- **Trace SSE stream doesn't push verdict updates separately.** The
  cascade verdicts arrive in the same `generated_answers` row that
  carries `grounding_score`, so the existing SSE poll picks them up on
  the next tick. If we want a dedicated `event: cascade` frame, that's
  a follow-up — the current shape works because cascade always finishes
  before persistence and before SSE next tick.
- **Frontend cascade panel doesn't show judge reasoning.** The column
  exists (`judge_reasoning`, populated by `LiteLLMJudge`), but UI
  surface is intentionally minimal for v1. A follow-up can expand the
  panel into a popover when the verdict is `fail`.
