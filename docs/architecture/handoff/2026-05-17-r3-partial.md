## Decisions made during R3

1. **`_PrecomputedEmbedder` shim instead of refactoring
   `PgvectorVectorSearch.search` to accept a query vector.** The shim is
   local to `InProcessRetrievalClient` and preserves the
   `VectorSearch` Protocol surface. Wider signature change can come
   later if a second caller wants to pass a vector directly.
2. **`HybridRetrievalResult.embedding_usage` is optional.** `bm25` mode
   never calls the embedder; the persistence stage falls back to the
   legacy zero-token row so cost dashboards keep joining cleanly on
   `query_session_id`.
3. **Idempotency key includes a body hash.** Spec says "hashed with
   `tenant_id`"; adding the body too is best-practice (a client
   shouldn't be able to recycle a key against a different payload),
   and the cost is one sha256 per request.
4. **Followers poll, don't 409.** A naive impl returns 409 Conflict on
   concurrent same-key requests; the user experience is much better
   when followers wait briefly and return the leader's result. 60s
   poll cap matches the 60s wall-clock query budget.
5. **No timeout retries.** `LiteLLMGenerator` no longer retries on
   `LiteLLMTimeout` / `TimeoutError`. Three retries at a 60s timeout
   == 180s before the user sees a failure, well past any sane request
   budget. Other exceptions keep the existing
   `retry_if_not_exception_type` behavior.
6. **Reservation TTL = `generation_timeout + 30s` cushion.** Auto-
   cleanup if the orchestrator crashes between reserve and release.
   The release path is idempotent so the cushion never under-counts.
7. **Redis-unavailable degrades both features to no-op rather than
   failing the request.** The route stays up in partial Redis outages;
   the trade-off is that the reservation gate temporarily can't see
   in-flight charges. The lifespan startup logs the degradation so
   operators know.

## Deferred to next session

### R3.S6 — Embedder hoist (only)
The plan listed five components to hoist. On inspection only `Embedder`
fits the "process-singleton" shape:

- **Embedder** — singleton candidate: model name + base URL are static
  at process startup.
- **Generator** — per-request: effective model can change (cloud vs
  ollama) so per-request `LiteLLMGenerator(...)` construction is
  inherent. The R3.S4 work already lifted the timeout out of the
  per-request hot path.
- **KeywordSearch / VectorSearch / HybridRetriever** — bound to the
  request-scoped SQLAlchemy session. Per-request construction is the
  correct lifetime, not a perf problem to solve.

The scope for the next session's S6:
1. `apps/api/app/lifecycle.py` — construct `LiteLLMEmbedder` at startup
   under `app.state.embedder`.
2. `apps/api/app/dependencies.py` — `EmbedderDep` + `get_embedder`.
3. `apps/api/app/api/v1/routes/query.py` — read embedder from
   `app.state` via DI; pass into `Orchestrator(embedder=...)`.
4. `apps/api/app/services/rag/orchestrator.py` — accept an embedder
   instance directly (drop `embedding_model` + `ollama_base_url`
   kwargs that drive per-request construction); the `embedder` field
   on `QueryContext` already exists.
5. Update the `FakeSettings` block in
   `apps/api/tests/unit/test_query_response_options.py` to inject the
   embedder.
6. Estimated effort: ~30 minutes.

## Open follow-ups surfaced during R3

- **Reservation cleanup on stuck releases.** The TTL covers a stuck
  release, but if the API process dies mid-request the reservation
  decays naturally — verified with the unit tests but not
  load-tested. Add a chaos drill in Phase R4 / R5 work.
- **Idempotency cache invalidation is purely TTL-driven.** If a tenant
  wants to re-run a query with the same `Idempotency-Key` (e.g. for
  testing), they need to wait 24h or rotate the key. That's the
  Stripe-style contract — document it in the OpenAPI description in a
  follow-up.
- **The `_FakeRedis` in the unit tests** is intentionally minimal.
  Integration tests against testcontainers Redis will exercise real
  `SCAN` cursor pagination + actual NX semantics.
