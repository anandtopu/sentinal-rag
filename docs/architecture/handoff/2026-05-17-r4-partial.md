## Decisions made during R4

1. **Env-var named `RETRIEVAL_TRANSPORT`, not `RETRIEVAL_MODE`.** The
   remediation plan called for `RETRIEVAL_MODE`. The codebase already
   carries `RETRIEVAL_MODE=hybrid` in `.env.example` as a per-request
   retrieval-mode default (which is intentionally *not* loaded into
   Settings). Reusing the name collided with Pydantic's Literal
   validator and broke 3 test collections. Documented in ADR-0031.
2. **Service-to-service auth is a shared bearer token.** Per the
   plan: "JWT verification (same Keycloak JWKS cache as the API)" is
   the production-grade target. For R4 v1 we ship a simpler shared
   token + leave the upgrade path documented in ADR-0031 §
   "Notes on the design docs." The NetworkPolicy + bearer token give
   us two layers of defense today.
3. **`_PrecomputedEmbedder` is duplicated, not imported.** The
   retrieval-service has its own copy of the small precomputed-
   embedder shim (vs. importing from `apps/api/`). Importing from
   `apps/api/` would create a cross-service import cycle; the
   structural protocol the embedder satisfies keeps the two copies
   in sync without a hard dependency.
4. **`/capabilities` advertises the upgraded surface only when
   `SERVICE_TOKEN` is set.** Without a token the route returns the
   diagnostic-only shape so unauthenticated probes can't enumerate
   the real backend list. The old tests pin the diagnostic shape;
   the new tests pin the upgraded shape under a `dependency_override`.
5. **Default `retrieval_transport` stays `in-process` until the
   benchmark.** The Helm chart's `values.yaml` flips it to `http` for
   deployed environments (so the workload exists end-to-end in K8s)
   but the Python default remains `in-process`. Local dev + unit
   tests keep the legacy library path until ADR-0029's harness-
   generated benchmark says the HTTP p95 is inside the SLO.

## Open follow-ups surfaced during R4

- **Settings.retrieval_service_token defaults to empty.** When
  `RETRIEVAL_TRANSPORT=http` *and* the token is empty, the API will
  send `Authorization: Bearer ` and the service refuses with 503.
  A startup-time guard could fail fast instead — punt to the next
  session.
- **HPA on retrieval at CPU 70 / mem 75% is copied from the API.**
  The retrieval workload is more CPU-bound (embedding work) than the
  API; the benchmark will inform whether to tune these.
- **`infra/terraform/` is untouched.** The retrieval workload is
  another pod on the existing EKS/GKE; no new Terraform module is
  needed (the plan called this out — verified by inspection).
  Per-cloud IRSA / WI binding for the retrieval ServiceAccount is a
  `values-{aws,gcp}-dev.yaml` overlay job — punt to the live-deploy
  phase.

## Risks the next session should pay attention to

1. **`RETRIEVAL_TRANSPORT=http` in the chart, `in-process` in Python
   defaults.** Anyone running `helm template ... | kubectl apply` to
   a real cluster will boot the API in HTTP mode and need both the
   retrieval pod and the bearer secret present. Local docker-compose
   + `uv run uvicorn` will boot in-process. This is intentional and
   documented in ADR-0031, but it's the most likely source of "why
   does my deploy 503" confusion.
2. **Bearer token is a single shared secret.** Anyone with read
   access to the API pod's secret can talk to retrieval directly.
   The NetworkPolicy mitigates this (only api pods can route there),
   but the auth model is the production upgrade path called out in
   ADR-0031.
3. **The retrieval-service has its own DB engine + session pool.**
   That's two pool counts to size in tuning. If you're running on a
   tight Postgres connection budget, halve `pool_size` on both
   services for the same effective total.
4. **The `_PrecomputedEmbedder` duplicate** in
   `apps/retrieval-service/.../main.py` mirrors the one in
   `apps/api/.../client.py`. They satisfy the same structural
   protocol; if you add a method to the in-process one, mirror it
   here.
