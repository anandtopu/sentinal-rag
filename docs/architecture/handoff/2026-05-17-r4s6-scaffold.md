## Decisions made during R4.S6

1. **No hand-edited numbers in the ADR.** The plan called this out
   (ADR-0029); the scaffold section in ADR-0031 deliberately has no
   tables, only pointers to where the live-run data lands. The data
   appears in `docs/operations/eval-report.md` + `docs/operations/retrieval-benchmark-report.md` once the runbook runs.
2. **Default transport stays `http` in `values.yaml`** (per ADR-0031),
   but the runbook documents the rule for flipping back to
   `in-process`: the HTTP path's p95 exceeding
   `SLO.queryP95Ms` (4000 ms today) is the only condition that triggers
   the flip.
3. **The R4.S6c runbook acknowledges one temporary hand-edit
   exception:** until the planned `summarize_transport.py` script
   lands, the operator computes the latency table by hand from k6
   console output. The runbook flags this explicitly as a temporary
   measure; the summarizer is implementation work outside R-stream
   scope.

## Risks the next session should pay attention to

1. **The k6 scenario requires a populated demo collection.** The
   `SENTINELRAG_COLLECTION_IDS` env var is mandatory; the runbook
   assumes `make seed` (or equivalent) has been run against both
   benchmark namespaces. A fresh cluster with no chunks will return
   abstentions and the latency comparison will be meaningless.
2. **Two Helm releases for the benchmark share Postgres.** If the
   ExternalSecret for one namespace points at a different DSN than
   the other, the quality comparison fails because retrieval sees
   different chunks per side. Runbook calls this out — verify the
   DSN match before launching k6.
3. **k6 thresholds in `transport-compare.js` mirror `baseline.js`.**
   If a future change tunes the baseline thresholds, the
   transport-compare scenario stays out of sync until updated.
   Better fix: import the threshold dict from a shared module (one-
   liner refactor, deferred).
